# Product

## Register

product

## Users

Rush UI 服务于构建后台、管理端和仪表盘 Web 应用的前端工程师、组件库维护者和产品研发团队。使用者通常在需要稳定交付业务界面、统一交互语义、降低重复实现成本时接入组件。

他们的核心任务是快速组装可访问、可维护、类型可靠的 React 管理端界面，同时保持组件 API 简洁、行为一致、样式可主题化。

## Product Purpose

Rush UI 是一个面向后台系统的 React 组件库 monorepo。它提供基础组件、设计 token、文档示例、测试和发布流程，让团队以统一的工程约定构建管理端产品。

成功状态是组件行为可靠、公共 API 稳定、文档和 Storybook 能解释真实用法，新增组件可以按同一套类型、无障碍、测试和样式约定持续扩展。

## Brand Personality

Rush UI 的品牌人格是简洁、创新、自信、有审美。它应该像成熟的产品工具一样克制可信，同时在细节、交互和视觉判断上体现清晰的设计品味。

语气应直接、专业、中文优先。组件和文档不需要营销式表达，应让使用者快速理解能力边界、状态行为和接入方式。

## Anti-references

Rush UI 不应呈现 AntD 克隆感。它可以采用管理端用户熟悉的交互模式，但不能在视觉语言、API 命名和文档表达上显得只是对现成企业组件库的复刻。

需要避免模板化 SaaS 视觉、过度玻璃拟态、堆叠卡片、过宽阴影和为了独特而牺牲可维护性的实验视觉。

## Design Principles

1. 熟悉但不复制：保留管理端组件的学习成本优势，同时在命名、密度、状态反馈和视觉节奏上建立 Rush UI 自己的判断。
2. 行为先于装饰：组件首先要在键盘、表单、状态、错误和加载场景下可靠工作，视觉增强不能破坏语义。
3. API 保持克制：公共接口只暴露稳定、高频、可解释的能力，避免把内部 DOM 和临时设计细节变成长期负担。
4. 文档展示真实场景：示例应覆盖后台系统里的表单、筛选、表格工具栏、操作按钮和状态边界，而不是只展示孤立外观。
5. 主题能力来自 token：色彩、间距、圆角、阴影和焦点样式应沉淀为可维护变量，让组件可扩展而不散落硬编码。

## Accessibility & Inclusion

没有额外指定的无障碍要求时，Rush UI 默认以可靠的管理端可访问性为底线：语义 HTML 优先，ARIA 只在需要时使用，交互组件支持键盘操作、清晰焦点态、disabled / loading / error 状态和可访问名称。

动画和状态反馈应尊重 `prefers-reduced-motion`。颜色不能成为唯一状态表达；文本、边框、图标或语义属性需要共同支撑错误、警告、成功和禁用等状态。
